# CipherGuard

A C++ security toolkit — password strength checking, strong password generation, and TOTP-based two-factor login — compiled to WebAssembly and wrapped in an interactive login + verification demo.

## What this project demonstrates

- **Password strength scoring** (length, character variety, common-password detection, repetition checks)
- **Strong password generation** using `<random>`
- **TOTP (Time-based One-Time Password) generation**, RFC 6238-style, built from scratch with a hand-written SHA1 and HMAC-SHA1 — no external crypto library
- The **same C++ logic** runs both as a terminal CLI and inside the browser via WebAssembly (Emscripten)
- A two-step **login → TOTP verification** flow, with the login ID accepted as either an email or a 10-digit phone number

## Part 1 — Run the CLI version

Requires a C++17 compiler (g++/clang).

```bash
g++ -std=c++17 -Wall -o cipherguard src/main.cpp
./cipherguard
```

## Part 2 — Build the WebAssembly version

### Install Emscripten (one-time setup)
```bash
git clone https://github.com/emscripten-core/emsdk.git
cd emsdk
./emsdk install latest
./emsdk activate latest
source ./emsdk_env.sh        # on Windows: emsdk_env.bat
```

### Compile the bindings
From the `cipherguard/` project root:
```bash
emcc src/wasm_bindings.cpp -o web/auth.js -s WASM=1 -lembind \
     -s MODULARIZE=1 -s EXPORT_NAME="AuthModule"
```
This produces `web/auth.js` and `web/auth.wasm`. Both need to be committed to your repo — GitHub Pages only serves static files, and these are the compiled engine the page depends on.

### Run it locally
Browsers block loading `.wasm` files directly from `file://` for security reasons, so you need a local server:

**Option A — VS Code Live Server extension:** right-click `web/index.html` → "Open with Live Server."

**Option B — Python's built-in server:**
```bash
cd web
python3 -m http.server 8000
```
Then open `http://localhost:8000` in your browser.

## Part 3 — How the login flow works

1. **Login screen** — enter an email or 10-digit phone number, then a password. The strength meter updates live as you type. The **Login** button only enables once the score is above 50%.
2. **TOTP screen** — re-enter the same login ID. If it matches what you logged in with, a 6-digit code is generated and shown on-screen in a "demo delivery" box (labeled as email or SMS depending on which ID type you used).
3. Enter the code within its 30-second window (shown by the countdown ring) to reach the **success screen**.

**On session persistence:** the logged-in ID is kept only in a JavaScript variable for the current page load. Refreshing the browser clears it completely — there's no localStorage, sessionStorage, or cookie involved, by design.

**On "demo delivery":** this project shows the OTP code directly on-screen instead of actually sending an SMS or email. Real delivery would require a backend server (to keep the OTP secret server-side) plus a paid SMS/email API (e.g., Twilio, SendGrid) — out of scope for a static, zero-cost, front-end demo, but straightforward to add on top of this same TOTP engine.

## Deploying to GitHub Pages

```bash
git init
git add .
git commit -m "Initial commit: CipherGuard"
git remote add origin https://github.com/YOUR_USERNAME/cipherguard.git
git branch -M main
git push -u origin main
```

Then on GitHub: **Settings → Pages → Source: branch `main`, folder `/web`.** Your live demo will be at `https://YOUR_USERNAME.github.io/cipherguard/`.

## Why this design

Rather than reimplementing the security logic twice (once in C++, once in JavaScript), the C++ code is compiled once to WebAssembly and used as the single source of truth for both the terminal tool and the website — so the CLI and the live demo are provably running identical logic.
